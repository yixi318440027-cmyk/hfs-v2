# HFS 二开产品路线图 v1.0

> 基于社区 Issue 分析、架构改进评估、文件管理页面设计、竞品对标及 frp-ext 协同机会的完整路线图。
> 制定日期：2026-06-23 | 评审状态：待 Stakeholder Alignment

---

## 一、产品愿景

> **打造「轻量但专业」的 Go 语言文件服务器**——在保持单二进制部署、10MB 级体积的前提下，提供对标 Cloudreve 的文件管理体验、Alist 的多协议支持、以及 Filestash 的连接灵活性。

**为什么是 Go 重写？**
- 内存占用从 50-150MB 压到 10-30MB
- 原生高并发（Goroutine），消除 Node.js 单线程瓶颈
- 单二进制部署，无运行时依赖
- 与 frp-ext 技术栈统一（Go + SQLite + Vue 3），代码复用、运维统一

对所有候选需求按 RICE = (Reach × Impact × Confidence) / Effort 打分。

### 打分尺度说明

| 维度 | 说明 |
|------|------|
| **Reach** | 覆盖用户数（占目标用户百分比） |
| **Impact** | 对核心指标影响：3=巨大, 2=显著, 1=中等, 0.5=轻微 |
| **Confidence** | 置信度：100%=数据支撑, 80%=社区验证, 50%=逻辑推断 |
| **Effort** | 人月（PM） |

## 三、Now / Next / Later 路线图

### 🟢 NOW（当前 → 2026 Q3）—— MVP 核心闭环

**目标**：Go 版本达到 HFS 功能对等 + 文件管理体验显著优于 HFS。

```
Phase 1：基础架构 + 文件浏览 MVP
├── Go HTTP Server（net/http + chi router）
├── VFS 虚拟文件系统（YAML 配置驱动）
├── YAML 配置解析 + SQLite 数据层
├── 账户认证（SRP 或 bcrypt，先最小实现）
├── 列表视图 + 面包屑导航 + 排序
├── 文件名模糊搜索（前端实时过滤）
├── 基础文件操作（下载、删除、重命名、新建文件夹）
├── 上传（拖拽 + 点击 + 进度条）
├── WebDAV 基础支持
└── 跨平台编译（Windows/Linux/macOS/Docker）
```

**非目标（Non-goals）**：
- 不做 FTP/SFTP
- 不做全文搜索
- 不做插件系统
- 不做移动端专属适配

**成功指标**：
| 指标 | 基线 | 目标 |
|------|------|------|
| 二进制体积 | HFS 22MB | <12MB |
| 内存占用（空闲） | HFS ~50MB | <20MB |
| 文件列表加载（1000 文件） | HFS ~800ms | <300ms |
| 并发下载（100 连接） | HFS 退化 | 吞吐量不退化 |
| 用户完成"查找→下载"时间 | - | <15 秒 |

---

### 🟡 NEXT（2026 Q4）—— 体验增强 + 差异化

**目标**：文件管理体验对标 Cloudreve，搜索能力形成差异化优势。

```
Phase 2：交互深度 + 搜索增强
├── 网格视图 + 缩略图自动生成
├── 右键上下文菜单 + 快捷键体系（Ctrl+C/V、Delete、F2、Space 预览）
├── 目录树侧栏（可拖拽移动文件）
├── 多选 + 批量操作（批量下载/删除/移动）
├── 搜索筛选（类型/时间/大小/目录范围）
├── SQLite FTS5 文件名+内容全文搜索
├── 拖拽上传文件夹（递归）
├── 分享链接（时效 + 密码 + 下载次数限制）
└── 存储空间概览（侧栏底部进度条）

Phase 3：协议扩展
├── HTTP/2 支持
├── FTP/FTPS 支持
└── IP 多地址绑定
```

**非目标（Non-goals）**：
- 不做 HTTP/3（QUIC 生态尚未成熟）
- 不做 S3 兼容 API（留到 Later）
- 不做 OAuth2/OIDC（留到 Later）

**成功指标**：
| 指标 | 目标 |
|------|------|
| 搜索响应时间（FTS5） | <50ms（10 万文件） |
| 用户操作效率（批量删除 50 文件） | <3 次点击 |
| 分享链接使用率 | >10% 活跃用户使用过 |
| FTP 下载速度（vs HTTP） | 不低于 HTTP 的 90% |

---

### 🔵 LATER（2027 Q1-Q2）—— 企业特性 + 生态

**目标**：覆盖企业用户场景，建立插件生态。

```
Phase 4：企业级能力
├── RBAC 权限模型（角色-权限-资源）
├── OAuth2/OIDC 集成（对接企业 SSO）
├── 审计日志（操作记录、登录记录）
├── 存储/流量配额管理
├── 回收站（可配置保留时间）
├── 文件在线预览（PDF/Office/代码高亮）
├── DDNS 集成（Cloudflare/AliDNS）
└── 仪表盘（流量趋势、热门文件、在线用户）

Phase 5：生态与多端
├── 移动端 PWA（滑动操作、底部 FAB）
├── 插件系统（Lua 脚本引擎或 Go plugin）
├── S3 兼容 API
├── 多节点管理（统一控制面板）
└── 文件版本历史
```

**非目标（Non-goals）**：
- 不做协同编辑（偏离定位）
- 不做 AI 文件分类/标签（需求不明确）
- 不做桌面客户端（PWA 覆盖）

## 七、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| GPL-3.0 许可证限制商用 | 中 | 高 | 评估是否接受 GPL，或参考设计独立实现 |
| WebDAV 兼容性测试成本高 | 高 | 中 | Phase 1 先支持基础 WebDAV，Phase 3 再做兼容性矩阵 |
| 功能范围蔓延（从文件服务器变网盘） | 中 | 高 | 每 Phase 明确 Non-goals，季度 review 砍需求 |
| Go 版插件生态不如 Node.js | 高 | 低 | Phase 5 前不投入插件，先做好核心 |
| frp-ext 代码耦合度高，复用困难 | 低 | 中 | Phase 1 抽取通用库（auth/db/log），独立 package |

## 八、成功度量体系

### 北极星指标
> **周活跃文件操作数（WAO）**：每周至少完成 1 次下载/上传/分享的用户数。

### L1 健康指标

| 分类 | 指标 | Phase 1 目标 | Phase 2+ 目标 |
|------|------|-------------|--------------|
| 获取 | 新部署实例数/月 | >100 | >500 |
| 激活 | 首次上传完成率 | >80% | >90% |
| 留存 | 30 天留存率 | >40% | >60% |
| 性能 | P95 搜索响应时间 | <300ms | <50ms |
| 性能 | 内存占用（空闲） | <20MB | <20MB |
| 满意度 | "查找→下载"耗时 | <15s | <10s |

### L2 诊断指标

- 文件列表首屏加载时间（P50/P95）
- 上传成功率 / 失败原因分布
- 搜索使用率（搜索会话数 / 总访问数）
- 分享链接创建数 / 访问数
- 右键菜单使用率（各操作项占比）

## 九、决策记录

| 决策 | 结论 | 原因 |
|------|------|------|
| 语言选型 | Go | 性能、部署、与 frp-ext 协同 |
| 数据库 | SQLite（modernc.org/sqlite） | 无 CGO、轻量、与 frp-ext 一致 |
| 前端框架 | Vue 3 + Vite | 与 frp-ext 一致，降低维护成本 |
| 全文搜索 | SQLite FTS5 | 轻量，无需外部服务 |
| 认证方案 | SRP（兼容 HFS 账户迁移）或 bcrypt | 先最小实现，Phase 4 再增强 |
| 许可证 | 待定（评估 GPL vs MIT） | 需法律评估后再确定 |
